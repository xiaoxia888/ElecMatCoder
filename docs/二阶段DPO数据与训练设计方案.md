# 二阶段 DPO 数据与训练设计方案

本文档用于设计本项目在一阶段 SFT 之后的二阶段偏好优化方案，目标是针对当前结构化抽取任务中常见的以下问题进行定向优化：

- 边界缺失
- 字段漏抽
- 字段幻觉
- schema 结构不稳定
- `STANDARD_*` 拆分或绑定错误

本文档面向当前项目主线任务：

- 输入：管道材料描述文本
- 输出：结构化 JSON

二阶段方案默认建立在以下前提上：

- 已有一阶段 SFT 模型
- 一阶段输出已基本具备可用性
- 当前主要问题不是“完全不会抽”，而是“抽得不够稳、不够守规矩”

## 1. DPO 在本项目中的定位

### 1.1 不是替代 SFT，而是补一层行为校正

本项目的一阶段任务属于强结构抽取任务，主能力仍应通过 SFT 学到：

- 学会字段定义
- 学会 JSON schema
- 学会尺寸、壁厚、标准拆分等基本模式

DPO 的作用不是替代这些基础能力，而是进一步优化输出偏好，使模型更倾向于：

- 宁缺勿猜
- 边界完整
- 字段归属正确
- schema 合法
- 多标准绑定准确

### 1.2 适合优化的问题

本项目二阶段 DPO 建议重点优化以下问题：

- 原文没有该字段，却输出了字段值
- 原文有字段，但只抽出一部分
- 输出格式不符合约定 schema
- `STANDARD` 主体和 `STANDARD_GRADE` / `STANDARD_METHOD` 没拆开
- 多个 `STANDARD` 存在时，`bind_to_index` 挂错
- 不确定时强行猜测字段值

### 1.3 不适合交给 DPO 单独解决的问题

以下问题不应主要依赖 DPO：

- 某类字段在训练集中本来就样本太少
- 模型从未学会某种抽取模式
- 规范本体知识覆盖不足
- 基础 JSON 生成能力太差

这些问题优先通过以下方式解决：

- 补充 SFT 数据
- 修正一阶段标注规范
- 增强规则校验和后处理

## 2. 本项目推荐的 DPO 总目标

建议将二阶段 DPO 的优化目标定义为：

**让模型在不确定时更保守，在确定时更完整，并严格服从结构化抽取 schema。**

该目标可以拆成四条可执行偏好：

1. 无证据不输出
2. 有证据就尽量抽全
3. 必须守 schema
4. 优先字段归属正确，而不是表面上“像对了”

## 3. 推荐数据格式

### 3.1 标准 DPO 样本格式

推荐使用标准偏好对格式：

```json
{
  "prompt": "<完整 system + user 或 user-only 输入>",
  "chosen": "<更优输出>",
  "rejected": "<较差输出>",
  "meta": {
    "error_type": "幻觉",
    "source": "线上案例",
    "difficulty": "中等"
  }
}
```

若后续采用 `trl` 的 `DPOTrainer`，通常更推荐拆成三列：

```json
{
  "prompt": "你是一个管道材料信息提取助手。...\\n用户输入: 90度弯头 DN50 SCH40 A234 WPB ASME B16.9",
  "chosen": "{\"TYPE\":\"90度弯头\",\"SIZE\":{\"DN\":[\"DN50\"]},\"THICKNESS\":{\"SCHEDULE\":[\"SCH40\"]},\"MATERIAL\":\"A234 WPB\",\"STANDARD\":\"ASME B16.9\"}",
  "rejected": "{\"TYPE\":\"弯头\",\"SIZE\":\"DN50\",\"MATERIAL\":\"WPB\"}"
}
```

### 3.2 本项目建议保留附加元信息

建议保留 `meta` 字段，便于后续抽样、分析和分桶训练：

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "...",
  "meta": {
    "task": "一阶段抽取",
    "error_type": "边界缺失",
    "field": "MATERIAL",
    "source": "模型预测",
    "sample_id": "ner_000123",
    "difficulty": "困难"
  }
}
```

推荐规则：

- `key` 保持简洁稳定，便于代码处理
- `value` 使用中文，便于人工整理、筛选和复核

推荐 `meta` 字段：

- `task`: 任务类型，当前建议写 `一阶段抽取`
- `error_type`: 错误类型
- `field`: 主要受影响字段
- `source`: 数据来源，如 `模型预测`、`自动扰动`、`人工整理`
- `sample_id`: 原始样本编号
- `difficulty`: 难度分级，如 `简单`、`中等`、`困难`
- `comment`: 可选，人审备注

## 4. prompt 设计建议

### 4.1 prompt 是否保留 system prompt

建议保留与 SFT 一致的 system prompt，不要换任务口径。

推荐：

- `prompt = system + user`

不建议：

- 只保留裸输入文本，导致训练时任务边界不一致

原因：

- 你当前一阶段 SFT 是在固定 system prompt 下训练的
- DPO 若换成另一种 prompt 形式，可能导致偏好学习和原任务分布脱节

### 4.2 user 内容保持原始输入

user 部分建议继续使用一阶段原始输入文本，不额外注入解释性标签。

原因：

- DPO 应对齐真实推理时输入
- 不要把“错误说明”写进 prompt，否则变成另一个任务

## 5. chosen / rejected 的构造原则

### 5.1 `chosen` 是什么

`chosen` 是你认可的更优输出，应满足：

- 与一阶段 schema 完全兼容
- 字段边界尽量完整
- 不输出原文中没有证据支持的字段
- 多标准拆分和绑定正确
- JSON 结构合法

### 5.2 `rejected` 是什么

`rejected` 不是随便乱造的垃圾答案，而应尽量接近模型真实会犯的错误。

推荐来源：

- 当前线上/离线模型真实输出错误
- 历史预测失败样本
- 基于正确答案自动扰动生成的“高相似错解”

不推荐：

- 构造与任务完全无关的胡乱字符串
- 构造一眼就极差的无效输出

原因：

- DPO 学的是偏好边界
- 如果 `rejected` 太差，训练信号价值有限

### 5.3 负样本要“错得像真的”

优先构造以下类型的 `rejected`：

- 少一个字段
- 某个字段边界截断
- 某个字段多猜一个值
- schema 近似正确但某个层级错了
- `bind_to_index` 错挂

这种“高相似错误”最适合 DPO。

## 6. 错误类型设计

建议为 DPO 数据定义统一错误类型标签。

### 6.1 一级错误类型

推荐使用以下一级类目，建议 `key` 保持 `error_type`，但 `value` 使用中文：

- `幻觉`
- `边界缺失`
- `字段缺失`
- `结构错误`
- `字段混淆`
- `绑定错误`
- `过度抽取`
- `抽取不足`

### 6.2 各错误类型定义

#### `幻觉`

定义：

- 原文没有明确信号，模型却输出了字段或字段值

示例：

- 原文没有压力等级，却输出 `PRESSURE = "CL150"`
- 原文没有标准，却猜了一个 `STANDARD`

#### `边界缺失`

定义：

- 字段抽到了，但边界不完整

示例：

- 正确：`ASTM A403 WP316L`
- 错误：`WP316L`

#### `字段缺失`

定义：

- 原文明确存在某字段，但模型完全没输出

示例：

- 原文有 `DN80xDN50`，却没有 `SIZE`

#### `结构错误`

定义：

- 输出结构不符合项目约定 schema

示例：

- `SIZE = "DN80xDN50"`，而不是对象结构
- `THICKNESS = "SCH40"`，而不是对象结构

#### `字段混淆`

定义：

- 内容抽出来了，但放错字段

示例：

- 把 `Series I` 放到 `STANDARD`
- 把 `THD` 放进 `MATERIAL`

#### `绑定错误`

定义：

- `STANDARD_*` 与 `STANDARD` 的绑定关系错误

示例：

- `Series I` 应绑定第二个 `STANDARD`，却绑定到了第一个

#### `过度抽取`

定义：

- 抽取过多，把不该并入字段的内容也并进来了

示例：

- `STANDARD = "GB/T19326 Series I"`

#### `抽取不足`

定义：

- 抽取不充分，保留信息不足

示例：

- `TYPE = "弯头"`，漏掉 `90度`

## 7. 面向本项目的重点 DPO 桶

建议优先构造以下 6 类偏好桶。

### 7.1 幻觉压制桶

目标：

- 降低无依据补字段

典型字段：

- `PRESSURE`
- `STANDARD`
- `MATERIAL`

示例：

```json
{
  "prompt": "你是一个管道材料信息提取助手...\\nPIPE S31603 DN80 SCH40S",
  "chosen": "{\"TYPE\":\"PIPE\",\"SIZE\":{\"DN\":[\"DN80\"]},\"THICKNESS\":{\"SCHEDULE\":[\"SCH40S\"]},\"MATERIAL\":\"S31603\"}",
  "rejected": "{\"TYPE\":\"PIPE\",\"SIZE\":{\"DN\":[\"DN80\"]},\"THICKNESS\":{\"SCHEDULE\":[\"SCH40S\"]},\"MATERIAL\":\"S31603\",\"PRESSURE\":\"CL150\"}",
  "meta": {
    "error_type": "幻觉",
    "field": "PRESSURE"
  }
}
```

### 7.2 边界完整桶

目标：

- 减少边界截断

典型字段：

- `MATERIAL`
- `STANDARD`
- `TYPE`

示例：

```json
{
  "prompt": "你是一个管道材料信息提取助手...\\nREDUCER ASTM A403 WP316L BW ASME B16.9 DN80xDN50",
  "chosen": "{\"TYPE\":\"REDUCER\",\"SIZE\":{\"DN\":[\"DN80\",\"DN50\"]},\"MATERIAL\":\"ASTM A403 WP316L\",\"STANDARD\":\"ASME B16.9\"}",
  "rejected": "{\"TYPE\":\"REDUCER\",\"SIZE\":{\"DN\":[\"DN80\",\"DN50\"]},\"MATERIAL\":\"WP316L\",\"STANDARD\":\"B16.9\"}",
  "meta": {
    "error_type": "边界缺失",
    "field": "MATERIAL"
  }
}
```

### 7.3 schema 服从桶

目标：

- 强化结构一致性

典型字段：

- `SIZE`
- `THICKNESS`

示例：

```json
{
  "prompt": "你是一个管道材料信息提取助手...\\nWELDOLET / MSS SP-83 / 6000LB / A182 F304 / FNPT DN80*DN25",
  "chosen": "{\"TYPE\":\"WELDOLET;FNPT\",\"SIZE\":{\"DN\":[\"DN80\",\"DN25\"]},\"PRESSURE\":\"6000LB\",\"MATERIAL\":\"A182 F304\",\"STANDARD\":\"MSS SP-83\"}",
  "rejected": "{\"TYPE\":\"WELDOLET;FNPT\",\"SIZE\":\"DN80*DN25\",\"PRESSURE\":\"6000LB\",\"MATERIAL\":\"A182 F304\",\"STANDARD\":\"MSS SP-83\"}",
  "meta": {
    "error_type": "结构错误",
    "field": "SIZE"
  }
}
```

### 7.4 规范拆分桶

目标：

- 强化 `STANDARD` 主体与修饰项拆分

典型字段：

- `STANDARD`
- `STANDARD_GRADE`
- `STANDARD_METHOD`
- `STANDARD_APPENDIX`

示例：

```json
{
  "prompt": "你是一个管道材料信息提取助手...\\n异径三通, 15CrMoG, SW, 300LB, GB/T19326 SERIAL I , DN25 x DN15",
  "chosen": "{\"TYPE\":\"异径三通;SW\",\"SIZE\":{\"DN\":[\"DN25\",\"DN15\"]},\"PRESSURE\":\"300LB\",\"MATERIAL\":\"15CrMoG\",\"STANDARD\":\"GB/T19326\",\"STANDARD_GRADE\":[{\"value\":\"SERIAL I\",\"bind_to_index\":0}]}",
  "rejected": "{\"TYPE\":\"异径三通;SW\",\"SIZE\":{\"DN\":[\"DN25\",\"DN15\"]},\"PRESSURE\":\"300LB\",\"MATERIAL\":\"15CrMoG\",\"STANDARD\":\"GB/T19326 SERIAL I\"}",
  "meta": {
    "error_type": "字段混淆",
    "field": "STANDARD"
  }
}
```

### 7.5 多标准绑定桶

目标：

- 提升 `bind_to_index` 正确率

示例：

```json
{
  "prompt": "你是一个管道材料信息提取助手...\\n管子 A/G S.S; 20 Galvanized; GB/T5310; HG20553(Sr B); φ159X5mm; NPS6",
  "chosen": "{\"TYPE\":\"管子\",\"SIZE\":{\"OD\":[\"159\"],\"INCH\":[\"6\\\"\"]},\"THICKNESS\":{\"MM\":[\"5\"]},\"MATERIAL\":\"20 Galvanized\",\"STANDARD\":[\"GB/T5310\",\"HG20553\"],\"STANDARD_GRADE\":[{\"value\":\"Sr B\",\"bind_to_index\":1}]}",
  "rejected": "{\"TYPE\":\"管子\",\"SIZE\":{\"OD\":[\"159\"],\"INCH\":[\"6\\\"\"]},\"THICKNESS\":{\"MM\":[\"5\"]},\"MATERIAL\":\"20 Galvanized\",\"STANDARD\":[\"GB/T5310\",\"HG20553\"],\"STANDARD_GRADE\":[{\"value\":\"Sr B\",\"bind_to_index\":0}]}",
  "meta": {
    "error_type": "绑定错误",
    "field": "STANDARD_GRADE"
  }
}
```

### 7.6 保守输出桶

目标：

- 强化“不确定就不输出”

适用场景：

- 文本噪声大
- 模糊缩写多
- 存在多个似是而非候选

这类样本很适合用来打压幻觉。

## 8. DPO 数据来源设计

本项目建议 DPO 数据采用混合来源。

### 8.1 来源 A：模型真实错误输出

优先级最高。

来源包括：

- 当前本地模型批量预测结果
- 线上接口错误案例
- 用户人工校正样本

优点：

- 与真实分布最贴近
- 错误最有代表性

推荐占比：

- `40% ~ 60%`

### 8.2 来源 B：基于金标自动生成负样本

从现有 `ner_data_new_schema.json` 金标出发，按规则构造 `rejected`。

可自动生成的负样本类型：

- 删除字段
- 截断字段边界
- 把 `SIZE` / `THICKNESS` 改回字符串
- 把 `STANDARD_GRADE` 塞回 `STANDARD`
- 随机错绑 `bind_to_index`
- 增加一个不存在字段值

优点：

- 规模可控
- 覆盖面大
- 易于分桶

推荐占比：

- `30% ~ 50%`

### 8.3 来源 C：人工精修偏好对

适合高价值难例：

- 多标准复杂绑定
- 边界难判断
- 中英混写复杂样本
- 类型与旧字段融合边界不清的样本

优点：

- 信号质量最高

推荐占比：

- `10% ~ 20%`

## 9. 自动构造 rejected 的建议规则

建议先做一个可控扰动器，从金标自动生成偏好对。

### 9.1 可用扰动算子

推荐的基础扰动算子：

- `drop_field`: 删除一个字段
- `truncate_value`: 截断字段值
- `flatten_structured_field`: 将对象字段压平为字符串
- `merge_standard_modifier`: 把 `STANDARD_*` 合并进 `STANDARD`
- `wrong_bind_index`: 改错 `bind_to_index`
- `add_fake_field`: 增加一个原文中没有的字段
- `move_field_value`: 把值放入错误字段

### 9.2 扰动强度控制

每条样本建议只施加 1 到 2 个错误，不建议一次改太多。

原因：

- 单错样本更容易学到明确偏好
- 多错混合样本不利于分析 DPO 的收益来源

推荐比例：

- 单错误：`70%`
- 双错误：`25%`
- 三错误及以上：`5%`

## 10. 数据配比建议

### 10.1 总量建议

起步阶段建议先做一个中等规模 DPO 集：

- `3k ~ 10k` 条偏好对

对你当前项目来说，优先做高质量，不急着过大。

### 10.2 错误类型配比建议

建议初始分布如下：

- `幻觉`: `25%`
- `边界缺失`: `20%`
- `结构错误`: `15%`
- `字段缺失`: `15%`
- `绑定错误`: `10%`
- `字段混淆`: `10%`
- `过度抽取` / `抽取不足`: `5%`

### 10.3 字段覆盖建议

建议重点覆盖：

- `STANDARD`
- `STANDARD_GRADE`
- `STANDARD_METHOD`
- `MATERIAL`
- `SIZE`
- `THICKNESS`
- `TYPE`

如果资源有限，建议优先做：

1. `STANDARD` 相关
2. `MATERIAL` 边界
3. `SIZE` / `THICKNESS` schema
4. 幻觉压制

## 11. 训练方案建议

### 11.0 一阶段与二阶段的分工

判断原则：

- 若错误属于高频、基础、结构性抽取失败，应优先在一阶段 SFT 修正
- 若错误属于一阶段已经大体会做，但仍偶发不稳、偏保守或偏激进，则适合放到二阶段 DPO 加固

像下面这类样本：

```json
{
  "input": "三通 SW 216-EL41-68 316L ASME B16.9 DN300 x DN150 S-80 x S-80",
  "chosen": {
    "TYPE": "三通;SW",
    "SIZE": {
      "DN": ["DN300", "DN150"]
    },
    "THICKNESS": {
      "SCHEDULE": ["SCH80"]
    },
    "MATERIAL": "316L",
    "STANDARD": "ASME B16.9"
  },
  "rejected": {
    "TYPE": "三通",
    "SIZE": {
      "DN": ["DN300"]
    },
    "THICKNESS": {
      "SCHEDULE": ["SCH80"]
    },
    "MATERIAL": "316L",
    "STANDARD": "ASME B16.9"
  },
  "meta": {
    "task": "一阶段抽取",
    "error_type": "抽取不足",
    "field": "TYPE,SIZE",
    "source": "模型预测"
  }
}
```

这类错误同时表现为：

- `TYPE` 少抽了 `SW`
- `SIZE.DN` 少抽了第二个值 `DN150`

建议处理方式：

- 如果这类问题在常见样本中频繁出现，优先补一阶段 SFT 金标和训练数据
- 如果一阶段模型已大体学会，但仍偶发漏一个修饰词或少一个尺寸值，再用二阶段 DPO 强化“有证据就尽量抽全”的偏好

结论：

- 这类错误的根因更偏向一阶段抽取能力问题
- 最优策略通常是“一阶段先修主能力，二阶段再做偏好加固”

### 11.0.1 判定总原则

判断一条错误该归一阶段还是二阶段，优先看它属于哪一类问题：

- 如果是“模型本身还不会”，归一阶段
- 如果是“模型已经会，但经常不稳”，归二阶段 DPO
- 如果是“规则/后处理更适合兜底”，不要硬塞给训练阶段

一句话判断：

- 一阶段解决“能力有没有”
- 二阶段解决“行为偏不偏”

### 11.0.2 优先归一阶段的错误

以下错误优先通过一阶段 SFT 数据和标注规范修正：

#### A. 高频基础抽取错误

特征：

- 常见字段经常抽不出来
- 主字段经常少值
- 不是偶发，而是成批出现

典型例子：

- `DN300 x DN150` 经常只抽一个尺寸
- `SW`、`THD`、`RTJ` 这类常见修饰信息经常漏掉
- `ASTM A403 WP316L` 经常只剩一半

处理建议：

- 优先补一阶段金标
- 检查 schema 和标注口径是否一致
- 提高相关模式在 SFT 中的覆盖度

#### B. 新模式/长尾模式根本没学会

特征：

- 某一类表达几乎完全不会抽
- 模型对这类模式没有稳定正确输出

典型例子：

- 新增某类 `STANDARD_METHOD`
- 很少见的 `BWG`
- 特殊 `TYPE` 组合写法

处理建议：

- 先补 SFT
- 不要指望 DPO 从零教会模型新能力

#### C. schema 主体不熟

特征：

- `SIZE` / `THICKNESS` 老是输出成字符串
- 多层结构经常乱
- JSON 主结构都不稳定

处理建议：

- 这是基础结构能力问题，优先归一阶段

#### D. 标注规范本身不清导致的错误

特征：

- 同类样本在金标中本来就不一致
- 模型输出混乱源于数据标准不统一

处理建议：

- 先修规范
- 再清洗或补充一阶段数据

### 11.0.3 更适合留给二阶段 DPO 的错误

以下错误更适合在一阶段已有基础能力后，用 DPO 做偏好加固：

#### A. 幻觉型错误

特征：

- 原文没有证据，却喜欢猜一个字段
- 模型“过于积极”

典型例子：

- 原文无压力，却输出 `PRESSURE`
- 原文无标准，却补一个 `STANDARD`

原因：

- 这类问题更像输出偏好问题，DPO 很适合压制

#### B. 偶发边界缺失

特征：

- 大部分时候能抽对
- 但在复杂样本里会偶发少一截

典型例子：

- `ASTM A182 F316L` 偶发只剩 `F316L`
- `GB/T19326 SERIAL I` 偶发只把 `SERIAL I` 丢掉

原因：

- 模型已经有能力，只是偏好不稳定
- 这时 DPO 的收益会比较明显

#### C. 保守或激进倾向不合适

特征：

- 模型有时太敢猜，有时又过于保守
- 你想把它往某种业务偏好上拉

典型目标：

- 宁缺勿猜
- 有证据时尽量抽全

#### D. 多候选时的偏好选择错误

特征：

- 模型对多个近似答案都“会一点”
- 但总是偏向你不想要的那种

典型例子：

- `STANDARD` 主体和等级没完全拆干净
- `bind_to_index` 偶发挂错

原因：

- 这类不是不会，而是“偏好错了”
- 适合 DPO

### 11.0.4 不建议主要靠 DPO 解决的错误

以下问题不要主要靠 DPO：

- 训练集里几乎没有覆盖的模式
- 一阶段压根不会的字段
- 标注标准本身混乱
- 规则后处理能稳定解决的问题

如果一个错误能通过以下方式稳定解决，优先考虑它们：

- 规则提取
- schema 校验
- 后处理修正
- SFT 补数据

### 11.0.5 针对你当前项目的实用判定表

| 错误现象 | 优先阶段 | 原因 |
|------|------|------|
| `DN300 x DN150` 经常只抽一个 | 一阶段 | 属于基础多值抽取能力不足 |
| `SW`、`THD` 经常漏并入 `TYPE` | 一阶段 | 属于字段定义和抽取能力问题 |
| `SIZE` 经常输出成字符串 | 一阶段 | schema 基础能力未学稳 |
| `PRESSURE` 经常凭空猜测 | 二阶段 DPO | 典型幻觉，适合偏好压制 |
| `STANDARD` 偶发带上 `Series I` | 二阶段 DPO | 主体与修饰项拆分偏好不稳 |
| `bind_to_index` 偶发挂错 | 二阶段 DPO | 多候选偏好问题 |
| 复杂材质边界偶发少一段 | 二阶段 DPO | 已有能力但边界不稳定 |
| 标注口径前后不一致 | 先修规范，再做一阶段 | 训练阶段无法弥补规范混乱 |

### 11.0.6 对“三通 SW ... DN300 x DN150 ...”这类错误的结论

像你举的这个例子：

- `TYPE` 少了 `SW`
- `SIZE.DN` 少了 `DN150`

它更像：

- 高频基础抽取不足
- 多值抽取能力不足
- `TYPE` 修饰信息并入规则没有学稳

因此判定为：

- **优先归一阶段**

更具体地说：

1. 如果这类错在常见样本里经常出现，就必须先补一阶段 SFT
2. 等一阶段把主能力补稳之后，二阶段 DPO 再拿这类错做“尽量抽全”的偏好强化

所以它不是“只能放一阶段”或“只能放二阶段”，而是：

- 主修在一阶段
- 加固在二阶段

### 11.0.7 如果没有真实模型错误，是否需要 `prepare_dpo_data`

不一定需要。

`prepare_dpo_data` 的意义通常只是以下几种：

- 冷启动阶段快速生成一批可用偏好对
- 验证 DPO 训练链路是否能正常跑通
- 在没有真实错误样本前做一版对照实验

如果你后续打算人工收集真实模型错误，那么更合理的路径是：

1. 先把一阶段主能力修稳
2. 用真实模型跑样本
3. 人工整理真实错误
4. 再构建正式 DPO 数据

也就是说：

- 你**不必**把 SFT 数据机械转换成 DPO 数据
- 真实模型错误样本通常比合成偏好对更有价值

### 11.0.8 DPO 训练后如何继续量化部署

当前二阶段 DPO 训练输出目录类似：

- `outputs/qwen3_dpo/run_xxx/final`

这里通常保存的是 **DPO 阶段的 LoRA adapter**，不是最终量化模型。

后续部署路径与一阶段 SFT 基本一致，只是多了一步“先合并 DPO adapter”：

1. 用 DPO 训练得到的 adapter 和基座模型进行合并
2. 得到新的 merged 模型目录
3. 再按原来的流程导出 GGUF / Ollama 模型

可以理解为：

- 一阶段：`base -> SFT LoRA -> merged`
- 二阶段：`SFT merged 或指定基座 -> DPO LoRA -> DPO merged -> GGUF/量化`

所以 `outputs/qwen3_dpo` 不是最终可部署量化模型目录，而是二阶段训练产物目录。

真正要继续量化的，应该是：

- DPO adapter 合并后的 merged 模型目录

部署思路不变：

- `merged -> GGUF -> quantize -> Ollama`

只不过合并对象从一阶段 SFT adapter，变成了二阶段 DPO adapter。

### 11.1 推荐训练顺序

建议采用以下顺序：

1. 一阶段 SFT
2. 一阶段模型离线跑全量或抽样预测
3. 构建 DPO 偏好对
4. 二阶段 DPO 训练
5. 用结构化指标和业务指标复测

### 11.2 二阶段训练输入形式

建议沿用一阶段任务口径：

- 相同 system prompt
- 相同 user 输入格式
- 相同 assistant 输出 schema

不要在 DPO 阶段改成另一种任务表达。

### 11.3 是否只做 NER DPO

当前建议先只做一阶段抽取任务的 DPO，不要同时混入编码任务。

原因：

- 抽取任务和编码任务错误模式不同
- 混训会降低偏好信号纯度

建议：

- 第一批 DPO 只做 `NER`
- 编码任务若后续也有稳定偏好需求，再单独建第二套 DPO 数据

## 12. 评估指标建议

二阶段 DPO 不建议只看 loss，建议增加任务级指标。

### 12.1 结构化指标

推荐统计：

- JSON 可解析率
- schema 合法率
- 字段存在率
- 多标准绑定正确率

### 12.2 抽取质量指标

推荐统计：

- 字段级精确率 / 召回率 / F1
- 边界完整率
- 幻觉率
- 漏抽率

### 12.3 业务导向指标

推荐统计：

- 送入编码器后的成功率
- `need_review` 比例变化
- `STANDARD` 相关错误占比变化

## 13. 数据集目录建议

建议未来将 DPO 数据独立存放，例如：

```text
data/pipe/dpo/
├── train.jsonl
├── val.jsonl
├── test.jsonl
├── schema.md
└── error_taxonomy.yaml
```

建议再细分原始来源：

```text
data/pipe/dpo_sources/
├── online_cases.jsonl
├── synthetic_negatives.jsonl
└── manual_hard_cases.jsonl
```

## 14. 一个推荐的最小落地版本

如果你想快速启动二阶段 DPO，建议先做一个 MVP。

### MVP 范围

只覆盖以下 4 类错误：

- 幻觉
- 边界缺失
- schema 错误
- `STANDARD` 拆分/绑定错误

### MVP 数据量

- `3000` 条左右偏好对即可启动

建议组成：

- 1500 条模型真实错例
- 1200 条自动扰动负样本
- 300 条人工精修难例

### MVP 成功标准

若以下指标有明显改善，就说明 DPO 值得继续投入：

- 幻觉率下降
- 边界完整率上升
- schema 合法率上升
- `STANDARD_*` 绑定正确率上升

## 15. 结论

对本项目来说，二阶段 DPO 的最佳定位不是“让模型更会抽”，而是：

- 让模型更少乱猜
- 让模型更完整抽取
- 让模型更守 schema
- 让多标准归属更稳定

建议你后续优先围绕以下主题构造偏好对：

1. 幻觉压制
2. 边界完整
3. schema 服从
4. `STANDARD` 拆分与绑定

只要这四类数据做好，DPO 对当前项目的提升通常就会比较明显。
