# 语义解析器数据集标注规范

本文档用于维护一阶段语义解析器训练数据。

目标不是训练“直接填槽”的模型，而是训练：

1. 从原文中定位证据片段
2. 判断证据片段的语义角色
3. 生成最终结构化决策结果

## 1. 数据文件组织

训练数据统一维护在 `data/pipe/semantic_parser/train/` 下。

- `parser_train_main.json`
  - 主训练集
  - 放常规样本
  - 这是基础分布，不要随便删空

- `parser_train_contrast.json`
  - 对比样本集
  - 放边界判别样本
  - 例如 `MATERIAL vs STANDARD`、`SIZE vs THICKNESS`

- `parser_train_noise.json`
  - 噪声样本集
  - 放 OCR、空格断裂、分隔符异常、简写误写

- `parser_train_hard.json`
  - 难例样本集
  - 放多字段复合、长句、多干扰项样本

评测数据维护在 `data/pipe/semantic_parser/eval/` 下。

- `parser_eval_frozen.json`
  - 冻结评测集
  - 不参与训练

- `parser_eval_hard.json`
  - 冻结难例评测集
  - 不参与训练

## 2. 统一数据结构

所有训练文件都使用同一种 schema，不因样本类别改变格式。

```json
{
  "input": "原始材料描述",
  "output": {
    "mentions": [],
    "semantics": [],
    "decisions": {}
  }
}
```

### 2.1 `input`

- 原始材料描述
- 尽量保持真实输入样式
- 不要为了标注方便重写原文

### 2.2 `output.mentions`

用于保存原文证据片段。

```json
{
  "id": "m1",
  "text": "GB/T19326(II)",
  "type": "STANDARD_TERM"
}
```

字段含义：

- `id`
  - 证据片段编号
  - 在当前样本内唯一即可

- `text`
  - 原文证据片段
  - 尽量贴近原文
  - 不要凭空生成原文中不存在的文本

- `type`
  - 证据片段粗分类
  - 不是最终字段
  - 枚举见 [语义解析器数据集枚举说明.md](/Users/guoxi/Desktop/workspace/NJNCC/python_code/ElecMatCoder/docs/语义解析器数据集枚举说明.md)

### 2.3 `output.semantics`

用于表达证据片段在当前上下文中的语义角色。

```json
{
  "mention_id": "m1",
  "semantic_tag": "STANDARD_COMPOSITE"
}
```

字段含义：

- `mention_id`
  - 对应 `mentions[].id`

- `semantic_tag`
  - 该证据片段在当前样本中的语义角色
  - 例如 `TYPE_BODY`、`MATERIAL_GRADE`、`STANDARD_COMPOSITE`

### 2.4 `output.decisions`

用于保存最终结构化决策结果。

这一层允许做结构化拆解和轻度规范化。

例如：

- 原文证据：`GB/T19326(II)`
- `mentions`：整体保留 `GB/T19326(II)`
- `decisions`：

```json
{
  "STANDARD": [
    {
      "BODY": "GB/T19326",
      "GRADE": "II"
    }
  ]
}
```

## 3. 标注总原则

### 3.1 `mentions` 尽量贴原文

正确：

```json
{"text": "S-10S×SCH40S", "type": "THICKNESS_TERM"}
{"text": "DN80xDN50", "type": "SIZE_TERM"}
{"text": "GB/T19326(II)", "type": "STANDARD_TERM"}
```

不推荐：

```json
{"text": "SCH10S", "type": "THICKNESS_TERM"}
{"text": "DN80", "type": "SIZE_TERM"}
{"text": "GB/T19326", "type": "STANDARD_TERM"}
```

上面这些值可以放在 `decisions`，但不适合作为原文证据。

### 3.2 `decisions` 允许结构化和轻度规范化

允许：

- `S-10S` -> `SCH10S`
- `GB/ T19326` -> `GB/T19326`
- `DN80xDN50` -> `["DN80", "DN50"]`

不允许：

- `NACE` 直接改成 `抗硫`
- `GALV` 直接改成 `镀锌`

这类属于知识解释，不属于原文结构化。

### 3.3 复合证据优先整体保留

如果原文本身是一个连续复合片段，优先整体保留到 `mentions`。

例如：

- `GB/T19326(II)`
- `DN150XDN80`
- `SCH40SXSCH40S`

然后在 `decisions` 里再拆解。

### 3.4 相同值重复出现时，不要在 `decisions` 去重

例如原文：

- `SCH40SXSCH40S`

应保留：

```json
"THICKNESS": {
  "SCHEDULE": ["SCH40S", "SCH40S"]
}
```

不要压缩成一个 `SCH40S`。

## 4. 各字段标注规则

### 4.1 TYPE

`mentions`

- 尽量保留原文主体证据
- 中文主体允许带修饰词，只要原文连续且确实属于种类描述

例如：

```json
{"text": "加强管嘴(短型) 弧底型", "type": "TYPE_TERM"}
```

`decisions`

- 只保留编码真正使用的主体

例如：

```json
"TYPE": {
  "BODY": "加强管嘴"
}
```

### 4.2 SIZE

优先保留带语义标识的原文片段。

例如：

- `DN80xDN50`
- `外径114mm`
- `φ108`
- `6\"x2\"`
- `L=300mm`

不推荐只留裸数字：

- `114`
- `108`

### 4.3 THICKNESS

优先保留原文壁厚证据。

例如：

- `S-10S×SCH40S`
- `SCH40SXSCH40S`
- `壁厚5.0mm`
- `4mmX4mm`

不推荐直接把 `mentions` 写成：

- `SCH10S`
- `5.0`

### 4.4 MATERIAL

材质相关证据建议拆成多个 `mention`，而不是一整段全塞成一个。

例如：

```json
{"text": "ASTM A403", "type": "MATERIAL_TERM"}
{"text": "WP316L", "type": "MATERIAL_TERM"}
{"text": "NACE", "type": "SPECIAL_REQ_TERM"}
```

`decisions` 再落到：

```json
"MATERIAL": {
  "RELATION": "single",
  "ITEMS": [
    {
      "EXEC_STANDARD": "ASTM A403",
      "GRADE": "WP316L",
      "SPECIAL_REQ": ["NACE"]
    }
  ]
}
```

### 4.5 STANDARD

规范类优先使用整体复合证据。

例如：

```json
{"text": "GB/T19326(II)", "type": "STANDARD_TERM"}
```

对应语义：

```json
{"mention_id": "m7", "semantic_tag": "STANDARD_COMPOSITE"}
```

`decisions` 再拆：

```json
"STANDARD": [
  {
    "BODY": "GB/T19326",
    "GRADE": "II"
  }
]
```

同理：

- `HG20553(Serial B)`
- `NB/T47010 Method E`
- `ASME B16.5 Appendix A`

都优先整体保留原文证据。

## 5. 新增样本如何分类维护

新增样本格式不变，仍然用同一种 schema。

区别只在于它主要解决什么问题。

### 放入 `parser_train_main.json`

适用：

- 普通常规样本
- 结构清晰
- 没有明显边界判别价值

### 放入 `parser_train_contrast.json`

适用：

- 专门解决边界判断问题

典型例子：

- `MATERIAL` 和 `STANDARD` 容易混
- `SIZE` 和 `THICKNESS` 容易混
- `TYPE.BODY` 与其他子标签边界不稳

### 放入 `parser_train_noise.json`

适用：

- 主要价值在于噪声鲁棒性

典型例子：

- `GA LVANIZED`
- `ANTT- SCC`
- `GB/ T19326`
- `1.名称: ... 2.材质: ...`

### 放入 `parser_train_hard.json`

适用：

- 长句
- 多字段复合
- 多标准、多材质、多干扰项
- 需要模型做复杂边界判断

## 6. 训练使用方式

训练时不直接只喂一个文件，而是训练前合并：

- `main`
- `contrast`
- `noise`
- `hard`

数据准备脚本：

```bash
python apps/trainer/qwen3_finetune/prepare_semantic_parser_data.py \
  --config apps/trainer/qwen3_finetune/config_semantic_parser.yaml
```

训练入口：

```bash
python -m apps.trainer.qwen3_finetune.train_semantic_parser \
  --config apps/trainer/qwen3_finetune/config_semantic_parser.yaml
```

## 7. 当前建议

对现有数据，不要再反向把 `parser_train_main.json` 硬拆成互斥四份。

正确做法是：

1. 保留当前主集完整不动
2. 后续新增样本开始按 `main/contrast/noise/hard` 分类维护
3. 冻结评测集单独维护，不参与训练
