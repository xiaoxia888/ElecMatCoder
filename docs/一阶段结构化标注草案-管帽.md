# 一阶段结构化标注草案-管帽

## 1. 目的

本文件只定义一件事：

- 管帽模型的一阶段结构化标注方式

这里的前提已经固定：

- 不同分类使用不同模型
- 一个模型一套结构

因此本文件只讨论：

- 管帽模型应该输出什么结构
- 管帽模型下 `BODY` 如何归一
- 管帽模型下 `TYPE` 哪些字段需要标注

---

## 2. 基本原则

### 2.1 一个模型一套结构

既然已经决定按分类拆模型，那么管帽模型只保留管帽真正需要的字段。

### 2.2 `BODY` 按编码种类归一

`BODY` 按**最终编码种类**归一。

原则：

- 编码相同，可以归一
- 编码不同，必须分开

当前管帽模型下，至少需要区分：

- `管帽`
- `封头`
- `堵头`

### 2.3 `TYPE` 字段标注分工

管帽模型下：

- `BODY`：标注管帽类编码种类
- `MANU`：标注制造方式
- `CONN`：标注连接方式
- `ENDS`：标注端部形式

例如：

- `CAP`
  - 应标成 `BODY=管帽`

- `PIPE CAP`
  - 应标成 `BODY=管帽`

- `封头`
  - 应标成 `BODY=封头`

- `堵头`
  - 应标成 `BODY=堵头`

---

## 3. 管帽模型的输出结构

当前建议管帽模型的一阶段输出中，`TYPE` 保留：

```json
"TYPE": {
  "BODY": "",
  "MANU": [],
  "CONN": [],
  "ENDS": []
}
```

说明：

- `BODY`：管帽类编码种类
- `MANU`：制造方式
- `CONN`：连接方式
- `ENDS`：端部形式

---

## 4. 管帽模型下 `BODY` 当前建议

当前至少保留以下标准 `BODY`：

- `管帽`
- `封头`
- `堵头`

### 4.1 `BODY` 完整归一化映射

以下为当前数据集中已出现写法到标准 `BODY` 的归一化结果。

#### 归一到 `管帽`

- `管帽` -> `管帽`
- `CAP` -> `管帽`
- `Cap` -> `管帽`
- `PIPE CAP` -> `管帽`

#### 归一到 `封头`

- `封头` -> `封头`

#### 归一到 `堵头`

- `堵头` -> `堵头`

---

## 5. 管帽模型下其他 `TYPE` 字段

### 5.1 `MANU`

以下制造方式进入 `MANU`：

- `SMLS`
- `WELDED`
- `ERW`
- `EFW`
- `SAWL`
- `SAWH`
- `FORGED`

补充规则：

- 原文出现 `SEAMLESS` 时，统一标为 `SMLS`
- 原文出现 `无缝` 时，如需要标制造方式，也统一标为 `SMLS`

### 5.2 `CONN`

以下连接方式进入 `CONN`：

- `SW`
- `SF`
- `NPT`
- `THD`

### 5.3 `ENDS`

以下端部形式进入 `ENDS`：

- `SO`
- `FNPT`
- `MNPT`
- `MTE`
- `TSE`
- `FTE`
- `NPTF`

## 7. 当前结论

管帽模型当前采用以下规则：

- `TYPE` 保留：
  - `BODY`
  - `MANU`
  - `CONN`
  - `ENDS`
- `BODY` 当前至少区分：
  - `管帽`
  - `封头`
  - `堵头`
- `CAP / Cap / PIPE CAP`
  - 当前统一归到 `管帽`
- `封头`
  - 当前单独保留为 `封头`
- `堵头`
  - 当前单独保留为 `堵头`
