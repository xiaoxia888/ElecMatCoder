# 当前编码方案思路与实现方法

## 1. 整体方案

当前材料编码流程采用两阶段方案：

1. 一阶段：从原始材料描述中提取结构化信息
2. 二阶段：将结构化结果转换为系统内部编码

目前两个阶段都基于 `Qwen3-4B`，采用 `QLoRA + SFT` 进行微调。训练数据统一整理为 `system + user + assistant` 的对话格式，使模型学习“输入 -> 输出”的映射关系。

采用两阶段方案，主要是因为原始材料描述往往同时混杂种类、尺寸、壁厚、压力、材质、规范等多类信息，而最终编码又是多个字段编码后的组合结果。如果直接一步生成最终编码，模型既要完成信息拆解，又要完成编码组合，问题容易混在一起，错误也难以定位。

例如，对 `JACKET EQUAL TEE, WELD, A234 WPB, ASME B16.9, DN200xDN100, SCH40XSCH40` 这类描述，如果一步直接输出最终编码，模型需要同时判断：

- `JACKET` 和 `EQUAL TEE` 的组合关系
- `WELD` 是否参与种类编码
- `DN200xDN100` 和 `SCH40XSCH40` 的位置对应关系
- `A234 WPB` 属于材质
- `ASME B16.9` 属于规范

如果拆成两阶段，则可以先在一阶段得到结构化结果，再在二阶段分别编码。这样做的好处是，错误可以明确区分为“一阶段结构化提取问题”还是“二阶段编码映射问题”。

## 2. 一阶段：信息提取

### 2.1 目标

一阶段的目标，是将原始材料描述解析为结构化结果。

当前将材料描述拆分为六个主标签：

1. `TYPE`：种类
2. `SIZE`：尺寸
3. `THICKNESS`：壁厚
4. `PRESSURE`：压力等级
5. `MATERIAL`：材质
6. `STANDARD`：规范

### 2.2 结构设计

当前一阶段输出结构如下：

```json
{
  "input": "",
  "output": {
    "TYPE": {
      "BODY": "",
      "MANU": "",
      "CONN": "",
      "SEAL": "",
      "ENDS": ""
    },
    "SIZE": {
      "DN": [],
      "OD": [],
      "INCH": [],
      "LENGTH": []
    },
    "PRESSURE": "",
    "THICKNESS": {
      "MM": [],
      "INCH": [],
      "SCHEDULE": [],
      "SERIES": [],
      "BWG": []
    },
    "MATERIAL": {
      "RELATION": "single",
      "ITEMS": [
        {
          "EXEC_STANDARD": "",
          "GRADE": "",
          "SPECIAL_REQ": []
        }
      ]
    },
    "STANDARD": [
      {
        "BODY": "",
        "GRADE": "",
        "APPENDIX": "",
        "METHOD": ""
      }
    ]
  }
}
```

字段含义可以概括为：

- `TYPE`：描述种类主体及其相关属性，如工艺、连接方式、密封面、端部形式
- `SIZE`：描述公称直径、外径、英制尺寸和长度
- `THICKNESS`：描述壁厚表达形式
- `PRESSURE`：描述压力等级
- `MATERIAL`：描述材质标准、牌号及特殊要求
- `STANDARD`：描述规范主体及其附加属性

### 2.3 为什么一阶段拆分得这么细

当前一阶段不仅拆成了六个主标签，还在每个标签下继续细分子字段。这样设计的原因是希望把不同语义角色尽量拆开，为后续编码提供更干净的中间结果。

具体考虑包括：

1. `TYPE` 中既有主体，又可能带工艺、连接方式、密封面、端部形式，如果不拆开，后续二阶段很难判断哪些词参与类型编码。
2. `SIZE` 和 `THICKNESS` 后续要参与规则换算和查表，如果一开始不分清 `DN`、`OD`、`INCH`、`SCHEDULE`、`SERIES`，后面会更难处理。
3. `MATERIAL` 和 `STANDARD` 也都存在主体和附加信息并存的情况，不拆开时容易互相污染。

例如：

1. `PIPE DN40 L=50 SCH10S 304` 中，如果长度不单独拆成 `LENGTH`，后续尺寸编码和壁厚换算很容易把 `L=50` 当成普通尺寸的一部分。
2. `ELBOW DIN EN 10217-7-BENDING RADIUS 3D DESIGN A-90 323.9x5.6 AISI 304L;DN300` 这类样本中，种类、规范、尺寸和材质本身就缠在一起。如果不细拆，模型很难稳定提取。

### 2.4 实际训练方式

一阶段训练时，采用 ChatML 形式：

- `system`：约束输出字段和 JSON 结构
- `user`：原始材料描述
- `assistant`：结构化 JSON

示例：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是一个管道材料信息提取助手。从描述中提取结构化信息，以JSON格式输出。..."
    },
    {
      "role": "user",
      "content": "三通, 304 GR.IGB/T13401, S13-EL15-78, SMLS, DN400 x DN200, 10S x 10S"
    },
    {
      "role": "assistant",
      "content": "{\"TYPE\": {\"BODY\": \"三通\"}, \"SIZE\": {\"DN\": [\"DN400\", \"DN200\"]}, \"THICKNESS\": {\"SCHEDULE\": [\"SCH10S\"]}, \"MATERIAL\": {\"RELATION\": \"single\", \"ITEMS\": [{\"GRADE\": \"304 GR.I\"}]}, \"STANDARD\": [{\"BODY\": \"GB/T13401\"}]}"
    }
  ]
}
```

从任务本质上看，一阶段是在做“原始材料描述 -> 结构化标签结果”的信息提取任务。

## 3. 二阶段：编码转换

### 3.1 目标

二阶段的目标，是将一阶段提取出的结构化字段进一步转换为标准编码。

当前做法是：

1. 从一阶段结果中取出各主标签的值进行整理
2. 对每个字段构造单独的编码输入
3. 送入微调后的二阶段模型进行编码
4. 再将各字段编码按照“种类、尺寸、壁厚、压力、材质、规范”的顺序拼接为最终结果

### 3.2 当前输入方式

当前二阶段不是直接吃原始材料描述，而是吃一阶段已经提取好的字段值。不同字段的输入构造规则不同。

例如：

- `TYPE` 会按 `BODY + MANU + CONN + SEAL + ENDS` 的顺序拼接后送入模型
- `STANDARD` 会按规范主体和附加属性整理后送入模型

示例训练数据：

```json
{
  "input": "偏心异径;BLE x TSE(MNPT)",
  "output": "REBMN"
}
{
  "input": "GB/T19326(Ia)",
  "output": "GBT19326Ia"
}
```

对应训练格式：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是一个管道材料编码助手。将实体原始值转换为标准编码，以JSON格式输出。直接输出JSON。"
    },
    {
      "role": "user",
      "content": "{\"TYPE\": \"90度弯头;SW\"}"
    },
    {
      "role": "assistant",
      "content": "{\"TYPE\": \"90ELS\"}"
    }
  ]
}
```

从任务本质上看，二阶段是在做“结构化字段值 -> 标准编码”的编码映射任务。

### 3.3 二阶段中的部分业务规则

二阶段并不只是简单映射，还包含一部分业务规则。

#### 3.3.1 连接方式和端部形式存在优先级

在 `TYPE` 编码中，不同连接方式和端部形式的组合并不是简单并列关系，而是存在业务上的主次差异。

例如：

1. 如果 `MTE` 和 `MNPT` 同时出现，则取 `MNPT` 对应的编码。
2. 如果 `FTE` 和 `FNPT` 同时出现，则取 `FNPT` 对应的编码。
3. 如果 `THREAD/THD` 与更具体的内外螺纹标签同时出现，则优先取更具体的标签。

当前实现不是在代码里先写显式裁决逻辑，再把结果送入模型，而是通过构造带有这类组合关系的训练样本，让二阶段模型学习不同组合情况下应输出哪个编码。

#### 3.3.2 规范字段在编码前会做格式化和归一

`STANDARD` 在二阶段编码时，也不是原样直接送入模型，而是会先经过一定的规则化处理。

例如：

1. 对规范做格式统一，如去掉年份信息。
   - 例如：`GB/T12459-2017(Ia)` 最终编码为 `GBT12459Ia`。
2. 根据自定义规则，为规范赋予不同用途或类别。
   - 例如区分“制造标准”“生产标准”等，再决定其最终编码方式。
3. 某些规范还会进一步映射为系统内部使用的目标标准，用于后续尺寸、壁厚等规则换算。

## 4. 当前整体流程

整体流程如下：

1. 输入原始材料描述
2. 一阶段模型提取结构化信息
3. 将结构化字段拆开，按字段分别送入二阶段
4. 二阶段模型返回各字段编码
5. 按系统规则拼接为最终编码结果

## 5. 当前存在的主要问题

### 5.1 微调和提示词，哪条路线更适合当前场景

当前项目实际采用的是“小模型微调”路线，即一阶段和二阶段都基于 `Qwen3-4B + QLoRA + SFT` 进行训练。但对于这类管道材料编码任务，后续是否还有必要继续投入在“小模型微调”这条路线，还是应该转向“直接通过提示词工程调用更强的大模型”，目前还不明确。

提出这个问题的原因是：

1. 当前项目已经投入了较多精力在数据集设计、字段拆分和微调训练上，但模型在一些复杂样本上仍然容易表现得更像分类器，而不是稳定的语义解析器。
2. 例如，在复杂工业文本场景中，仅通过继续补数据和微调小模型来逼近目标，是否真的比直接使用更强大模型并通过提示词约束输出格式更值得投入，目前仍值得讨论。

因此，目前需要进一步判断：

1. 当前任务更适合走“小模型微调”路线，还是“提示词工程 + 更强大模型”路线。
2. 如果继续走小模型微调路线，哪些问题值得继续靠补数据解决。

### 5.2 一阶段仍可能更像分类，而不是充分理解原文

虽然一阶段已经能够输出结构化结果，但在一些复杂场景下，模型仍然可能更倾向于直接给出最终字段分类结果，而不是充分依据原文中的证据关系进行判断。

### 5.3 一阶段数据集是否有必要拆分得这么细

当前一阶段的字段拆分已经比较细，这有助于后续处理，但也带来了新的问题：

1. 字段拆得越细，标注难度和一致性要求越高。
2. 某些子字段边界本身就不稳定，例如 `TYPE.BODY`、`MANU`、`ENDS` 在复杂表达里经常互相缠绕。
3. 如果结构设计过细，但模型不能稳定学会这些边界，细粒度结构本身也可能成为负担。

例如：

- `不锈钢管件承插焊管接台, Surface Ra=1.6um DN600 x DN25 THD C1500 S30403 GR.III NB/T47010MSS SP-97` 这类样本中，种类主体、连接方式、尺寸、材质、规范高度缠绕，细拆是否真的有利于模型学习，目前仍需验证。

因此，目前还需要进一步确认：一阶段是否有必要继续保持当前这么细的拆分粒度，还是应该适当收敛。

### 5.4 二阶段规则复杂，尤其是 `TYPE`

二阶段中，`SIZE`、`THICKNESS`、`PRESSURE` 相对规则明确，但 `TYPE`、`MATERIAL`、`STANDARD` 更复杂。

尤其 `TYPE` 的问题最明显：

- 主体词和修饰词容易混在一起
- 工艺、端部、连接方式、附加说明都可能混入主体表达
- 同一个种类，不同修饰组合可能对应不同编码

因此，`TYPE` 不是简单词典映射问题，而更像组合解析问题。

### 5.5 二阶段完全依赖模型时稳定性不足

如果二阶段全部依赖模型：

- 复杂字段容易受上下文影响
- 某些字段本来规则明确，却仍然变成概率输出
- 出现错误时不如规则和查表容易定位

## 6. 目前希望请教的问题

目前最希望进一步请教和确认的问题有：

1. 当前这种“两阶段：提取 + 编码”的总体方案是否合理
2. 对当前这类任务而言，更推荐继续投入“小模型微调”，还是改为“提示词工程 + 更强大模型”
3. 一阶段是否有必要继续保持当前这么细的字段拆分粒度，还是应该适当收敛
4. 一阶段的数据结构和标注方式，是否还有更适合提升解析能力的设计
5. 二阶段是否应该按字段复杂度拆分技术路线，而不是所有字段都统一使用模型
6. 对 `TYPE` 这类高复杂字段，是否有更合适的建模或编码策略

## 7. 当前总结

当前系统的核心思路是：

1. 用一阶段模型完成原始描述到结构化结果的转换
2. 用二阶段模型完成结构化字段到标准编码的映射
3. 将复杂问题拆开处理，而不是一步完成全部任务

这套方案目前已经能够运行，并且在工程上具备一定可维护性。但在实际效果上，一阶段的解析稳定性和二阶段复杂字段的编码方式，仍然需要进一步优化和验证。
